# 整合層：含金量彙整與 Product Map

## Product Map

---

## UserManagementModule

**功能：**
    - 使用者帳號管理服務。
    - 使用者資料備份。
**目的：**
    - 管理使用者資料，用於資料備份或分析。
**做法：**
    - Firebase 集中管理。
**排除：**
    - 其他第三方社交登入。
**利弊：**
    - Apple ID 登入，Google 帳號登入，並存時會有資料衝突問題需要解決。故先不實作 Apple ID 登入。
    - Apple ID 登入時，使用者可以選擇不分享 Email，造成帳號合併困難。
    - Firebase Authentication 支援 Google 帳號登入，免費使用。
    - 使用者資料備份使用量高機率在 Firebase Storage 免費額度內。

---

## AccountingAppModule

**功能：**
    - 本地記帳、雲端同步、自動化規則的完整行動端產品。
**目的：**
    - 提供從交易記錄到自動化分類的完整帳務閉環，是整個產品飛輪的核心。
**做法：**
    - Local-First 架構，WatermelonDB 本地資料庫；付費版透過 Firestore 跨裝置同步。
**排除：**
    - Web 主控台 WebConsoleModule、AI 顧問 AIAdvisorModule、企業總體資料 MacroDataModule。
**利弊：**
    - Local-First 確保 100% 離線可用，但跨裝置同步需付費解鎖。
    - 行動端優先讓記帳場景覆蓋廣，但進階大螢幕操作需另開 Web 主控台。

---

### AccountingCoreModule — 本地記帳核心

**功能：**
    - 免費本地記帳功能的完整集合。
**目的：**
    - 建立可信任的財務紀錄基礎，是整個產品飛輪的起點，所有免費功能在此交付。
**做法：**
    - 操作即寫入 WatermelonDB，UI 立即重新渲染；App 啟動時自動補產生定期交易。
**排除：**
    - 雲端同步 CloudSyncModule、付費授權 PaymentModule。
**利弊：**
    - 完整離線能力讓記帳不受網路限制，但跨裝置同步需升級付費。
    - 預設帳戶與類別降低上手門檻，但無法完全符合所有使用者分類習慣。

---

#### CoreRecording — 核心記錄

**功能：**
    - 支出、收入、轉帳三種交易類型的記錄與管理。
    - 帳戶管理，含預設帳戶建立。
    - 類別管理，含預設類別建立。
**目的：**
    - 提供完整的財務資料輸入閉環，讓使用者建立可信任的底層帳務資料。
**做法：**
    - WatermelonDB 本地資料庫，所有 CRUD 立即寫入；跨幣別轉帳自動記錄當下匯率。
**排除：**
    - 銀行 API 直接讀取交易資料。
**利弊：**
    - 手動輸入保有最高彈性與隱私，但入帳速度慢於自動匯入。
    - 跨幣別轉帳自動記錄匯率，但僅支援 Strict Direct Rate，不支援中間跳轉換算。

---

##### TransactionEditor — 交易編輯器

**功能：**
    - 建立、編輯、軟刪除支出、收入、轉帳三種交易。
    - 刪除後 4 秒內可 Undo 還原。
**目的：**
    - 作為使用者輸入財務資料的主要操作入口。
**做法：**
    - 停用與刪除功能統一在 Editor 管理，不使用 List 畫面的左滑操作；Undo 採用立即寫入 + 反向操作策略。
**排除：**
    - List 畫面的左滑刪除。
**利弊：**
    - 統一 Editor 操作避免手勢衝突，且職責劃分清晰；但相比左滑操作多一個點擊步驟。

---

##### AccountManagement — 帳戶管理

**功能：**
    - 建立、編輯、停用帳戶。
    - 設定帳戶幣別。
**目的：**
    - 讓使用者將資產對應到真實的銀行帳戶、現金或投資帳戶。
**做法：**
    - 本地資料庫儲存；停用不刪除，歷史交易保留關聯。
**排除：**
    - 帳戶餘額自動從銀行拉取。
**利弊：**
    - 軟停用保留歷史資料完整性，但停用帳戶數量累積後介面需過濾處理。

---

##### CategoryManagement — 類別管理

**功能：**
    - 建立、編輯、停用類別。
    - 預設類別集隨新帳戶自動建立。
**目的：**
    - 讓使用者依個人消費習慣組織財務分類，支撐報表統計準確度。
**做法：**
    - 預設資料集於初始化時自動建立；自訂類別付費 Tier 1 後可無限新增。
**排除：**
    - AI 自動分類建議，屬 AIAdvisorModule 負責範疇。
**利弊：**
    - 預設類別加速上手，但使用者仍需手動調整以符合個人習慣。
    - 免費版帳戶與類別數量有上限，需升級 Tier 1 解鎖無限制。

#### Presentation — 呈現介面

**功能：**
    - 首頁儀表板整合資產概覽與交易清單。
    - 配色主題系統，含內建主題集。
    - 交易 Note 欄位全文搜尋。
**目的：**
    - 降低使用者取得財務資訊的認知成本，並透過視覺設計降低記帳焦慮。
**做法：**
    - theme.ts 集中管理設計 Token，禁止 hardcode 顏色或字體；搜尋基於 WatermelonDB 本地索引。
**排除：**
    - 使用者自訂調色盤；跨裝置搜尋索引同步。
**利弊：**
    - 主題集統一視覺品質，但無法個人化調色。
    - 本地搜尋速度快且不洩露資料，但跨裝置搜尋索引無法同步。

##### HomeDashboard — 首頁儀表板

**功能：**
    - 期間資產餘額概覽與圓餅圖。
    - 交易清單，支援依類別或日期雙模式分組。
    - 時間區間切換，支援滑動手勢。
**目的：**
    - 讓使用者一眼掌握期間財務狀況。
**做法：**
    - 所有資料來自本地 WatermelonDB；時間邊界依使用者設定的主要時區計算。
**排除：**
    - 即時雲端資料更新。
**利弊：**
    - 本地計算速度快，但多裝置的資料需同步後才能一致顯示。

##### ThemeSystem — 配色主題

**功能：**
    - 主題切換介面。
    - 內建 4 套主題：Light Blue、Dark Blue、Ocean、Forest Green。
**目的：**
    - 透過視覺設計降低記帳焦慮，增加日常使用黏著度。
**做法：**
    - theme.ts 集中管理設計 Token，覆蓋顏色、字體、間距，禁止 hardcode。允許後續以小更新方式新增主題。
**排除：**
    - 使用者自訂調色盤。
**利弊：**
    - 固定主題集確保視覺品質一致，但無法滿足個人化需求。

##### SearchFunction — 交易搜尋

**功能：**
    - 對交易 Note 欄位的本地全文搜尋。
**目的：**
    - 讓使用者快速定位特定交易紀錄，無需依賴精確分類。
**做法：**
    - WatermelonDB 本地搜尋索引，純離線運作。
**排除：**
    - 跨帳本搜尋；雲端搜尋索引同步。
**利弊：**
    - 純本地速度快且隱私，但跨裝置搜尋索引無法同步。
    - 僅搜尋 Note 欄位，不支援搜尋金額或類別名稱。

#### DataTools — 資料工具

**功能：**
    - 定期交易的自動產生機制。
    - CSV 資料匯入流程。
**目的：**
    - 減少重複手動輸入，並降低從其他工具遷移的阻力。
**做法：**
    - App 啟動時執行 `generateMissingInstances()`，依 `lastRecurringCheckDate` 補產生遺漏的定期交易；CSV 匯入配合欄位映射確認介面。
**排除：**
    - 直接對接銀行 API 匯入交易。
**利弊：**
    - 本機觸發確保離線環境可用，但長期未開 App 時會一次補產生大量紀錄。

##### RecurringTransactions — 定期交易

**功能：**
    - 定期交易範本的建立、編輯、刪除，支援僅此筆或此筆之後兩種範圍。
    - App 啟動時自動補產生遺漏的定期交易實例。
**目的：**
    - 讓固定支出與收入自動記錄，減少重複手動輸入。
**做法：**
    - `generateMissingInstances()` 於 App 啟動時執行，依 `lastRecurringCheckDate` 對照當日時區補產生所有遺漏交易；定期交易補產生優先於批次同步執行。
**排除：**
    - 複雜條件式定期規則，例如每月最後一個工作日。
**利弊：**
    - App 啟動觸發確保離線環境也能自動產生，但長期未開 App 時會一次補產生大量紀錄。
    - 訂閱過期但未連網的使用者仍可短暫獲得新定期交易，此漏洞評估影響極低故接受。

##### DataImport — 資料匯入

**功能：**
    - CSV 檔案匯入流程。
    - 欄位映射確認介面。
**目的：**
    - 降低從其他記帳工具切換的阻力，縮短資料遷移成本。
**做法：**
    - CSV 解析後呈現互動式欄位映射介面，使用者確認對應關係後寫入 WatermelonDB。
**排除：**
    - 直接對接銀行 API 或第三方金融資料服務。
**利弊：**
    - CSV 格式普及性高，相容大多數記帳工具的匯出格式。
    - 需使用者手動對應欄位，若來源格式差異大則體驗較差。

### PaymentModule — 付費授權

**功能：**
    - 訂閱狀態驗證。
    - 序號兌換介面。
    - 後台序號管理。
**目的：**
    - 付費授權基礎設施，所有 Tier 付費牆的前置依賴。
**做法：**
    - RevenueCat 管理跨平台訂閱狀態；本地 TTL 快取 expirationDate 離線驗證授權；`checkPremiumStatus()` 本地計算，`refreshPremiumStatus()` 觸發網路更新。
**排除：**
    - 其他第三方支付渠道。
**利弊：**
    - RevenueCat 簡化跨平台訂閱管理，但引入第三方依賴。
    - TTL 快取讓離線使用者維持授權，但取消訂閱後存在短暫延遲。

### CloudSyncModule — 雲端同步

**功能：**
    - Firestore 批次雙向同步。
    - 手動立即同步觸發。
    - 多幣別帳戶與手動匯率管理。
    - Tier 1 解鎖無限帳戶與類別。
**目的：**
    - 讓付費用戶的跨裝置資料可信，並解鎖碎片資產的多幣別管理能力。
**做法：**
    - Delta batch sync，分三階段：Upload → Download → Finalize；Last-Write-Wins 衝突解決；手動同步 5 分鐘冷卻；Firestore 每日讀寫上限以本地計數管控，避免 Quota Tax。
**排除：**
    - 即時同步；外部匯率 API。
**利弊：**
    - 批次策略大幅降低 Firestore 費用，但使用者需手動或等待自動觸發。
    - 本地 Quota 計數防止單裝置失控，但無法限制多裝置的全域總量 Global Limit，評估可接受。
    - Firestore 廠商鎖定，未來遷移至自建後端成本較高。

---

#### SyncEngine — 同步引擎

**功能：**
    - 本地與 Firestore 的雙向 Delta 批次同步。
    - 衝突解決策略管理。
**目的：**
    - 以最低 Firestore 讀寫成本，確保付費使用者的跨裝置資料最終一致。
**做法：**
    - Upload 階段推送本地修改；Download 階段拉取雲端變更；Finalize 階段確認完成並更新 `lastSyncedAt`；以 `updatedOn` 欄位實作 Last-Write-Wins。
**排除：**
    - Firestore real-time listener 即時監聽。
**利弊：**
    - LWW 簡單可預測，但多裝置同時編輯同一筆資料時，較早的修改會靜默被覆蓋。

---

##### BatchSync — 批次同步機制

**功能：**
    - Delta 差量上傳與下載，僅傳輸 `lastSyncedAt` 之後有異動的資料。
    - 手動立即同步觸發，含 5 分鐘冷卻。
    - 升級 Tier 1 時執行首次完整同步 Initial Sync。
**目的：**
    - 最小化 Firestore 讀寫次數以控制費用，同時提供使用者主動同步的控制感。
**做法：**
    - 以 `updatedOn` 欄位篩選 Delta；降級 Tier 時停止 Sync Engine，保留本地資料。
**排除：**
    - 自動即時同步。
**利弊：**
    - Delta 同步大幅減少讀寫，但 schema 異動需謹慎維護版本相容性。

---

##### ConflictResolution — 衝突解決策略

**功能：**
    - Last-Write-Wins 衝突解決，縮寫 LWW，以 `updatedOn` 時間戳決定勝出版本。
**目的：**
    - 提供簡單可預期的衝突解決機制，避免需要使用者手動介入合併。
**做法：**
    - 同步時比對本地與雲端 `updatedOn`，保留較新的版本覆寫較舊的。
**排除：**
    - 三向合併 Three-way merge 或使用者手動合併介面。
**利弊：**
    - LWW 邏輯簡單且對使用者透明，但多裝置同時修改同一筆時，較早的改動會靜默遺失。

---

#### MultiCurrency — 多幣別記帳

**功能：**
    - 外幣帳戶建立。
    - 跨幣別轉帳，自動記錄當下匯率。
    - 手動匯率管理介面。
**目的：**
    - 讓碎片化的外幣資產得以統一管理，以基礎貨幣呈現跨幣別報表。
**做法：**
    - Strict Direct Rate：僅使用外幣對基礎貨幣的直接匯率計算報表，不支援中間跳轉；匯率由使用者手動輸入或由跨幣別轉帳自動記錄。
**排除：**
    - 外部即時匯率 API。
**利弊：**
    - 手動匯率避免 API 費用，且可反映私下換匯的實際匯率。
    - Strict Direct Rate 不支援跳轉，若無外幣到基礎貨幣的直接匯率，該筆資產在報表中無法計算。

---

##### CurrencyAccounts — 外幣帳戶

**功能：**
    - 建立幣別不同的帳戶。
    - 外幣帳戶在報表中依現行匯率換算為基礎貨幣顯示。
**目的：**
    - 讓持有多種外幣資產的使用者可以集中管理所有帳戶。
**做法：**
    - 每個帳戶綁定幣別；多幣別報表以基礎貨幣為統一計算基準；所有報表不回溯資產當時價格，使用現行匯率。
**排除：**
    - 帳戶餘額即時匯率更新。
**利弊：**
    - 靜態匯率計算簡單，但不反映市場即時價值波動。

---

##### ExchangeRateManagement — 匯率管理

**功能：**
    - 手動輸入並儲存外幣對基礎貨幣的匯率。
    - 跨幣別轉帳時自動從轉帳金額推算並記錄匯率。
**目的：**
    - 為多幣別報表提供換算基準，同時維持完整離線運作能力。
**做法：**
    - 僅儲存外幣對基礎貨幣的直接匯率；App 不限制使用者輸入的匯率值，支援私下換匯情境。
**排除：**
    - 外部匯率 API 自動更新。
**利弊：**
    - 完全離線可用，且可反映使用者實際換匯成本。
    - 使用者需自行維護匯率，若疏於更新則報表數字失真。

---

#### CapacityExpansion — 容量擴充

**功能：**
    - Tier 1 訂閱解鎖無限帳戶與類別數量。
**目的：**
    - 移除免費版的數量上限，讓進階使用者可建立完整的帳戶與類別架構。
**做法：**
    - RevenueCat 訂閱狀態驅動本地 PremiumContext；免費版數量上限以本地計數管控，主要目的是防止無窮迴圈等 Bug 造成單裝置失控。
**排除：**
    - 免費版漸進式釋放帳戶數量的設計。
**利弊：**
    - 二元解鎖設計，有限與無限兩種狀態，實作簡單，但降低免費用戶的漸進升級誘因。
    - 本地計數防止單裝置失控，但無法限制多裝置的全域總量，評估可接受。

---

### LogicEngineModule — 規則引擎

**功能：**
    - 規則式自動分類交易。
    - 時光機非破壞性歷史重算。
**目的：**
    - 讓使用者以規則批次處理交易分類，消除大量手動維護負擔。
**做法：**
    - 填空式規則編輯，條件觸發後自動套用；時光機以非破壞方式重算歷史資料。
**排除：**
    - AI 自動建議規則，屬 AIAdvisorModule 負責範疇。
**利弊：**
    - 規則式系統行為可預測且用戶可控，但需使用者手動建立規則，初期學習成本存在。
    - 非破壞性重算保護原始紀錄，但增加儲存與運算量。

---

#### RuleEngine — 規則管理

**功能：**
    - 規則中心，列出所有規則並可開關。
    - 填空式規則編輯器，支援條件與動作的組合設定。
**目的：**
    - 讓使用者定義若交易符合條件 A 則套用分類 B 的自動化邏輯。
**做法：**
    - 規則按優先順序排列；新交易進入時逐一比對規則並套用第一個匹配項。
**排除：**
    - AI 自動建立規則；複雜巢狀條件。
**利弊：**
    - 填空式介面降低設定門檻，但靈活度不如程式碼式規則引擎。

---

##### RuleCenter — 規則中心

**功能：**
    - 規則清單管理：建立、編輯、刪除、開關規則。
    - 規則優先順序排列。
**目的：**
    - 提供使用者管理所有自動化規則的統一入口。
**做法：**
    - 本地儲存規則集；規則開關不影響歷史已套用的分類結果。
**排除：**
    - 規則版本歷史記錄。
**利弊：**
    - 規則開關靈活，但關閉規則不會自動還原歷史套用結果，需手動觸發時光機重算。

##### RuleEditor — 規則編輯器

**功能：**
    - 填空式條件設定，支援金額、關鍵字、帳戶等欄位。
    - 填空式動作設定，支援套用類別、備註等動作。
**目的：**
    - 讓非技術使用者也能設定自動化分類規則。
**做法：**
    - UI 提供預設欄位組合，條件與動作各自為結構化表單。
**排除：**
    - 正規表達式或程式碼式條件輸入。
**利弊：**
    - 填空式設計易用，但表達能力有限，無法處理複雜條件。

#### TimeMachine — 時光機

**功能：**
    - 非破壞性歷史交易重算。
**目的：**
    - 讓使用者在修改規則後，可以將新規則套用至歷史資料，而不破壞原始紀錄。
**做法：**
    - 在副本上依新規則重算後回寫結果；原始交易資料不被覆蓋。
**排除：**
    - 破壞性覆蓋原始交易資料的重算方式。
**利弊：**
    - 非破壞性設計保護資料完整性，但增加儲存需求與重算運算量。

##### RecalculationEngine — 歷史重算

**功能：**

    - 支援選擇性重算，可指定時間範圍或全部。
**目的：**
    - 讓規則調整能回溯生效，提升歷史報表的準確度。
**做法：**
    - 在隔離副本計算後再回寫，確保重算過程中 UI 仍可正常使用。
**排除：**
    - 即時重算所有歷史資料，受效能限制不支援。
**利弊：**
    - 隔離計算保障 UI 不阻塞，但大量歷史資料時重算時間較長。
