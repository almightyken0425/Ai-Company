## Platform: App — 行動端 React Native

---

### Auth / AppClient — 身份驗證用戶端

- **功能：**
    - Google 帳號登入與登出。
    - 首次登入時自動建立預設帳戶與類別。
- **目的：**
    - 提供使用者統一身份，作為雲端同步與付費授權的基礎。
- **做法：**
    - Firebase Authentication 管理 Google 登入狀態；首次登入後自動 seed 預設帳戶與類別至本地 WatermelonDB。
- **排除：**
    - Apple ID 登入——資料衝突問題，暫不實作；其他第三方社交登入。
- **利弊：**
    - Firebase Authentication 支援 Google 帳號登入，免費使用。
    - Apple ID 登入時使用者可選擇不分享 Email，造成帳號合併困難，故先排除。

---

#### LoginFlow — Google 登入流程

- **功能：**
    - Google Sign-In 介面。
    - Firebase Authentication 登入與登出。
- **目的：**
    - 讓使用者透過已有的 Google 帳號快速完成登入，無需額外註冊流程。
- **做法：**
    - React Native Firebase SDK 整合；登入後 Firebase ID Token 作為後續 Firestore 存取憑證。
- **排除：**
    - 電子郵件加密碼登入；Apple ID 登入。
- **利弊：**
    - Google Sign-In 普及率高且免費，但不支援 Apple ID 的使用者需另有帳號策略。

---

#### PostAuthSetup — 首次登入初始化

- **功能：**
    - 首次登入時自動建立預設帳戶與類別集。
- **目的：**
    - 降低新使用者上手門檻，讓記帳可立即開始。
- **做法：**
    - `seed.ts` 於 AuthContext 偵測到新使用者後觸發，寫入 WatermelonDB。
- **排除：**
    - 使用者自訂初始資料集。
- **利弊：**
    - 預設資料集加速上手，但使用者仍需手動調整以符合個人習慣。

---

### RecordingCore — 記錄核心

- **功能：**
    - 支出、收入、轉帳三種交易類型的建立、編輯、軟刪除。
    - 定期交易範本管理與自動補產生。
    - 帳戶合併與類別合併。
    - 全域操作撤銷，支援 4 秒倒數還原。
    - CSV 資料匯入與匯出。
- **目的：**
    - 使用者寫入帳本的操作集合——包含所有建立、修改、刪除、補正財務紀錄的行為。
- **做法：**
    - 所有寫入即時落地 WatermelonDB；Undo 採立即寫入加反向操作策略；定期交易補產生於 App 啟動時執行，優先於批次同步。
- **排除：**
    - 銀行 API 直接讀取交易資料；AI 自動分類建議，屬 Cloud Service / AIAdvisor 範疇。
- **利弊：**
    - 本地寫入確保離線環境可正常記帳，但跨裝置一致需升級付費同步。
    - Undo 提供操作安全感，但 4 秒時間窗口關閉後無法還原，需配合批次操作謹慎使用。

---

#### TransactionEditor — 交易編輯器

- **功能：**
    - 建立、編輯、軟刪除支出與收入兩種交易類型。
    - 刪除後 4 秒內可 Undo 還原。
- **目的：**
    - 作為使用者輸入財務資料的主要操作入口。
- **做法：**
    - 停用與刪除功能統一在 Editor 管理，不使用 List 畫面的左滑操作；Undo 採用立即寫入加反向操作策略。
- **排除：**
    - List 畫面的左滑刪除。
- **利弊：**
    - 統一 Editor 操作避免手勢衝突，且職責劃分清晰；但相比左滑操作多一個點擊步驟。

---

#### TransferEditor — 轉帳編輯器

- **功能：**
    - 建立、編輯、軟刪除帳戶間轉帳。
    - 跨幣別轉帳時自動從金額推算並記錄隱含匯率。
- **目的：**
    - 讓使用者記錄資金在帳戶間的流動，並自動維護跨幣別的匯率歷史。
- **做法：**
    - 轉帳模型儲存 amountFrom、amountTo 與 impliedRate；跨幣別轉帳完成後自動建立對應的 CurrencyRate 記錄。
- **排除：**
    - 多步跳轉換算——Strict Direct Rate 策略，不支援中間幣別。
- **利弊：**
    - 自動記錄隱含匯率降低維護成本，但僅支援直接匯率，若無外幣對基礎貨幣直接匯率則無法換算報表。

---

#### RecurringTransactions — 定期交易

- **功能：**
    - 定期交易範本的建立、編輯、刪除，支援僅此筆或此筆之後兩種範圍。
    - App 啟動時自動補產生遺漏的定期交易實例。
- **目的：**
    - 讓固定支出與收入自動記錄，減少重複手動輸入。
- **做法：**
    - `generateMissingInstances` 於 App 啟動時執行，依 `lastRecurringCheckDate` 對照當日時區補產生所有遺漏交易；定期交易補產生優先於批次同步執行。
- **排除：**
    - 複雜條件式定期規則，例如每月最後一個工作日。
- **利弊：**
    - App 啟動觸發確保離線環境也能自動產生，但長期未開 App 時會一次補產生大量紀錄。
    - 訂閱過期但未連網的使用者仍可短暫獲得新定期交易，此漏洞評估影響極低故接受。

---

#### MergeOperation — 合併操作

- **功能：**
    - 帳戶合併：將來源帳戶的所有交易與轉帳遷移至目標帳戶後軟刪除來源。
    - 類別合併：將來源類別的所有交易遷移至目標類別後軟刪除來源。
    - 合併後 4 秒內可 Undo 還原。
- **目的：**
    - 讓使用者整理重複或多餘的帳戶與類別，維護資料結構整潔。
- **做法：**
    - `mergeService.ts` 批次更新關聯紀錄的 accountId 或 categoryId；自我轉帳的邊緣情況——來源與目標為同一帳戶——自動軟刪除以避免冗餘紀錄。
- **排除：**
    - 合併後自動去重複交易紀錄。
- **利弊：**
    - 批次遷移保留完整歷史紀錄，但合併後若需拆回只能透過 Undo，不支援事後分割。

---

#### UndoSystem — 撤銷系統

- **功能：**
    - 全域操作撤銷，4 秒倒數自動消失，支援手動提前關閉。
    - 支援交易、轉帳、帳戶、類別、合併五種撤銷類型。
- **目的：**
    - 提供操作安全網，降低使用者因誤操作造成的資料損失。
- **做法：**
    - `UndoContext` 管理全域 Undo 狀態；每次寫入操作完成後將反向操作登記至 Undo Stack；FloatingActionBar 顯示 Undo 按鈕。
- **排除：**
    - 多層級歷史撤銷；已完成同步的操作無法撤銷。
- **利弊：**
    - 立即撤銷的體驗直覺易用，但 4 秒時間窗口關閉後無法還原，需配合合併等批次操作謹慎使用。

#### DataManagement — 資料管理

- **功能：**
    - CSV 資料匯入流程。
    - CSV 資料匯出。
- **目的：**
    - 降低從其他工具遷移的阻力，並讓使用者可將資料帶出至外部工具。
- **做法：**
    - CSV 解析後呈現互動式欄位映射介面；匯出由前端直接產生，不依賴後端服務。
- **排除：**
    - 直接對接銀行 API 或第三方金融資料服務；自動排程匯出。
- **利弊：**
    - CSV 格式普及性高，相容大多數記帳工具的匯出格式。
    - 匯入需使用者手動對應欄位，若來源格式差異大則體驗較差。

##### DataImport — 資料匯入

- **功能：**
    - CSV 檔案匯入流程。
    - 欄位映射確認介面。
- **目的：**
    - 降低從其他記帳工具切換的阻力，縮短資料遷移成本。
- **做法：**
    - CSV 解析後呈現互動式欄位映射介面，使用者確認對應關係後寫入 WatermelonDB。
- **排除：**
    - 直接對接銀行 API 或第三方金融資料服務。
- **利弊：**
    - CSV 格式普及性高，相容大多數記帳工具的匯出格式。
    - 需使用者手動對應欄位，若來源格式差異大則體驗較差。

##### DataExport — 資料匯出

- **功能：**
    - 將交易清單匯出為 CSV 格式。
- **目的：**
    - 讓使用者可將資料帶出至外部工具進行進一步分析或備份。
- **做法：**
    - 前端直接產生匯出檔案，不依賴後端服務。
- **排除：**
    - 自動排程匯出；PDF 格式，屬 Web Console 功能。
- **利弊：**
    - 手動匯出簡單易用，但需用戶主動操作，缺乏自動化。

### AssetManagement — 資產結構管理

- **功能：**
    - 帳戶的建立、編輯、停用，每個帳戶綁定幣別代碼。
    - 類別的建立、編輯、停用，支援支出與收入兩種類型。
    - 外幣對基礎貨幣匯率的手動維護，以及幣別顯示格式設定。
    - 免費版帳戶與類別數量上限管控。
- **目的：**
    - 管理帳戶與類別——記帳的組織骨架——以及附著於帳戶的多幣別能力。
- **做法：**
    - 帳戶與類別採軟停用，歷史交易保留關聯；每個帳戶綁定幣別，多幣別報表以基礎貨幣統一換算；匯率由使用者手動輸入或由跨幣別轉帳自動記錄。
- **排除：**
    - 外部即時匯率 API；帳戶餘額自動從銀行拉取。
- **利弊：**
    - 軟停用保留歷史資料完整性，但停用帳戶數量累積後介面需過濾處理。
    - 手動匯率避免 API 費用且可反映私下換匯的實際匯率，但使用者需自行維護，疏於更新則報表失真。

#### AccountCRUD — 帳戶管理

- **功能：**
    - 建立、編輯、停用帳戶。
    - 設定帳戶幣別。
- **目的：**
    - 讓使用者將資產對應到真實的銀行帳戶、現金或投資帳戶。
- **做法：**
    - 本地資料庫儲存；停用不刪除，歷史交易保留關聯。
- **排除：**
    - 帳戶餘額自動從銀行拉取。
- **利弊：**
    - 軟停用保留歷史資料完整性，但停用帳戶數量累積後介面需過濾處理。

#### CategoryCRUD — 類別管理

- **功能：**
    - 建立、編輯、停用類別。
    - 預設類別集隨首次登入自動建立。
- **目的：**
    - 讓使用者依個人消費習慣組織財務分類，支撐報表統計準確度。
- **做法：**
    - 預設資料集於初始化時自動建立；自訂類別付費 Tier 1 後可無限新增。
- **排除：**
    - AI 自動分類建議，屬 Cloud Service / AIAdvisor / Backend 負責範疇。
- **利弊：**
    - 預設類別加速上手，但使用者仍需手動調整以符合個人習慣。
    - 免費版帳戶與類別數量有上限，需升級 Tier 1 解鎖無限制。

#### ExchangeRateManagement — 匯率管理

- **功能：**
    - 手動輸入並儲存外幣對基礎貨幣的匯率。
    - 跨幣別轉帳時自動從轉帳金額推算並記錄匯率。
    - 報表中以基礎貨幣統一呈現所有外幣帳戶餘額。
- **目的：**
    - 為多幣別報表提供換算基準，同時維持完整離線運作能力，並可反映使用者實際換匯成本。
- **做法：**
    - Strict Direct Rate：僅使用外幣對基礎貨幣的直接匯率計算報表，不支援中間跳轉；匯率採追加記錄策略，可透過新增記錄修正歷史。
- **排除：**
    - 外部匯率 API 自動更新。
- **利弊：**
    - 完全離線可用，且可反映使用者實際換匯成本。
    - 使用者需自行維護匯率，若疏於更新則報表數字失真。
    - Strict Direct Rate 不支援跳轉，若無外幣到基礎貨幣的直接匯率，該筆資產在報表中無法計算。

#### CurrencyDisplayConfig — 幣別顯示設定

- **功能：**
    - 設定各幣別的小數位數顯示。
    - 設定各幣別是否使用千位單位顯示。
- **目的：**
    - 讓不同幣別依其慣例正確呈現金額格式，提升閱讀準確性。
- **做法：**
    - `currency_configs` 資料表儲存每位使用者對每種幣別的顯示偏好；`PreferenceContext` 提供 `getCurrencyConfig` 供全 App 統一取用。
- **排除：**
    - 自動依 ISO 4217 標準套用格式——使用者可自訂覆蓋。
- **利弊：**
    - 使用者可依個人習慣調整顯示格式，但需主動設定，預設值可能不符所有幣別慣例。

#### PremiumFeatureGate — 付費功能閘控

- **功能：**
    - 免費版帳戶上限 3 個、類別上限 10 個的本地計數管控。
    - Tier 1 訂閱解鎖無限帳戶與類別數量。
- **目的：**
    - 透過資產結構的數量上限，驅動免費用戶升級付費。
- **做法：**
    - `premiumLogic.ts` 在帳戶或類別 CRUD 操作前檢查 PremiumContext tier；本地計數管控主要目的是防止無窮迴圈等 Bug 造成單裝置失控。
- **排除：**
    - 免費版漸進式釋放帳戶數量的設計。
- **利弊：**
    - 二元解鎖設計實作簡單，但降低免費用戶的漸進升級誘因。
    - 本地計數防止單裝置失控，但無法限制多裝置的全域總量，評估可接受。

### HomeDashboard — 首頁儀表板

- **功能：**
    - 期間資產餘額概覽與圓餅圖。
    - 交易清單，支援依類別或日期雙模式分組。
    - 時間區間與帳戶篩選，支援多種時間粒度切換。
    - 交易 Note 欄位全文搜尋。
- **目的：**
    - 讓使用者一眼掌握期間財務狀況，並快速定位特定紀錄。
- **做法：**
    - `HomeContext`、`HomeFilterContext`、`PeriodDataStore` 分別管理頁面狀態、篩選狀態、跨期間資料快取；快取最近 15 期避免重複計算；所有資料來自本地 WatermelonDB。
- **排除：**
    - 即時雲端資料更新；跨裝置搜尋索引同步。
- **利弊：**
    - 本地計算速度快，但多裝置的資料需同步後才能一致顯示。
    - 期間分頁避免一次載入過多紀錄，但跨期間查詢需切換多個頁面。

---

#### PeriodView — 期間分頁

- **功能：**
    - 以期間為單位的分頁瀏覽，支援滑動手勢切換。
    - 最近 15 期資料快取，避免重複計算。
- **目的：**
    - 讓使用者流暢瀏覽不同時間區間的財務狀況。
- **做法：**
    - `PeriodDataStore` 管理快取；時間邊界依使用者設定的主要時區計算；currentPeriodOffset 追蹤當前檢視的時間偏移量。
- **排除：**
    - 即時雲端資料更新。
- **利弊：**
    - 快取大幅提升滑動流暢度，但快取失效時機需謹慎處理，避免顯示過期資料。

---

#### FilterSystem — 篩選系統

- **功能：**
    - 時間粒度切換：日、週、月、年、全部。
    - 帳戶篩選，可選擇顯示特定帳戶的資料。

### CloudSync / AppClient — 雲端同步用戶端 — Tier 1

- **功能：**
    - Firestore 批次雙向同步，App 端。
    - 手動立即同步觸發。
    - Firestore 配額本地管控。
- **目的：**
    - 讓付費用戶的跨裝置資料可信，以最低 Firestore 讀寫成本實現最終一致。
- **做法：**
    - Delta batch sync，分三階段：Upload → Download → Finalize；Last-Write-Wins 衝突解決；手動同步 5 分鐘冷卻；Firestore 每日讀寫以本地計數管控，避免 Quota Tax。
- **排除：**
    - 即時同步；外部匯率 API。
- **利弊：**
    - 批次策略大幅降低 Firestore 費用，但使用者需手動或等待自動觸發。
    - 本地 Quota 計數防止單裝置失控，但無法限制多裝置的全域總量，評估可接受。
    - Firestore 廠商鎖定，未來遷移至自建後端成本較高。

---

#### SyncEngine — 同步引擎

- **功能：**
    - 本地與 Firestore 的雙向 Delta 批次同步。
    - 衝突解決策略管理。
- **目的：**
    - 以最低 Firestore 讀寫成本，確保付費使用者的跨裝置資料最終一致。
- **做法：**
    - Upload 階段推送本地修改；Download 階段拉取雲端變更；Finalize 階段確認完成並更新 `lastSyncedAt`；以 `updatedOn` 欄位實作 Last-Write-Wins。
- **排除：**
    - Firestore real-time listener 即時監聽。
- **利弊：**
    - LWW 簡單可預測，但多裝置同時編輯同一筆資料時，較早的修改會靜默被覆蓋。

---

##### BatchSync — 批次同步機制

- **功能：**
    - Delta 差量上傳與下載，僅傳輸 `lastSyncedAt` 之後有異動的資料。
    - 手動立即同步觸發，含 5 分鐘冷卻。
    - 升級 Tier 1 時執行首次完整同步 Initial Sync。
- **目的：**
    - 最小化 Firestore 讀寫次數以控制費用，同時提供使用者主動同步的控制感。
- **做法：**
    - 以 `updatedOn` 欄位篩選 Delta；降級 Tier 時停止 Sync Engine，保留本地資料。
- **排除：**
    - 自動即時同步。
- **利弊：**
    - Delta 同步大幅減少讀寫，但 schema 異動需謹慎維護版本相容性。

---

##### ConflictResolution — 衝突解決策略

- **功能：**
    - Last-Write-Wins 衝突解決，縮寫 LWW，以 `updatedOn` 時間戳決定勝出版本。
- **目的：**
    - 提供簡單可預期的衝突解決機制，避免需要使用者手動介入合併。
- **做法：**
    - 同步時比對本地與雲端 `updatedOn`，保留較新的版本覆寫較舊的。
- **排除：**
    - 三向合併 Three-way merge 或使用者手動合併介面。
- **利弊：**
    - LWW 邏輯簡單且對使用者透明，但多裝置同時修改同一筆時，較早的改動會靜默遺失。

---

#### QuotaManagement — 配額管控

- **功能：**
    - 本地計數追蹤每日 Firestore 讀寫次數。
    - 超出每日上限時暫停同步，防止單裝置 Quota Tax。
- **目的：**
    - 確保單一裝置不會因為異常操作耗盡 Firestore 免費配額，保護整體服務穩定性。
- **做法：**
    - `quotaService.ts` 本地計數；每日上限設定：2000 reads / 2000 writes per device；計數在 Firestore Sync 操作前後更新。
- **排除：**
    - 多裝置全域配額統計，評估可接受。
- **利弊：**
    - 本地計數實作簡單，但無法限制多裝置的全域總量，存在少量 Quota 漏洞，評估可接受。

### Authentication — Firebase 身份驗證

- **功能：**

- **排除：**

- **利弊：**
    - Firebase Authentication 免費使用，且與 Firestore 安全規則深度整合。
    - Apple ID 登入時使用者可選擇不分享 Email，造成帳號合併困難。

### Storage — Firebase 儲存

- **功能：**
    - 使用者資料備份。
- **目的：**
    - 管理使用者資料，用於資料備份。
- **做法：**
    - Firebase Storage 集中管理；備份使用量高機率在免費額度內。
- **排除：**
    - 對外提供資料下載 API。
- **利弊：**
    - Firebase Storage 免費額度通常足夠個人資料備份使用量。
    - 廠商鎖定問題同 Firestore，未來遷移成本存在。

---

### Firestore — 雲端資料庫

- **功能：**
    - 存放付費用戶的跨裝置同步資料，為 App / CloudSync / AppClient 的後端。
    - Firestore Security Rules 存取控制。
- **目的：**
    - 作為 Delta batch sync 的雲端資料來源，確保付費使用者的跨裝置資料最終一致。
- **做法：**
    - Collections 對應 WatermelonDB 表格：accounts / categories / transactions / transfers / currency_rates / schedules / users；Security Rules 確保使用者只能讀寫自己的資料。
- **排除：**
    - Real-time listener，基於成本控制不使用；App 端以外的直接 Firestore 寫入。
- **利弊：**
    - Firestore 無需維護伺服器，但廠商鎖定，未來遷移至自建後端成本較高。
    - 免費層每日配額 50k reads / 20k writes，project 層級，需搭配 App 端本地計數管控。

---

#### DataCollections — 資料 Collections

- **功能：**
    - 定義各 Firestore collection 的 schema 與欄位。
- **目的：**
    - 作為 App / CloudSync 的雲端資料模型，對應 WatermelonDB 本地 schema。
- **做法：**
    - Schema 定義於 `coreSchema.ts`；各實體包含 `updatedOn` 衝突解決時間戳與 `deletedOn` 軟刪除標記兩個欄位。
- **排除：**
    - 即時 listener subscription；伺服器端 business logic，僅負責資料儲存。
- **利弊：**
    - 純資料儲存設計簡單，但 schema 異動需謹慎維護版本相容性。

---

#### SecurityRules — 安全規則

- **功能：**
    - 使用者只能讀寫屬於自己的 collections 資料。
- **目的：**
    - 確保多租戶資料隔離，防止使用者存取他人資料。
- **做法：**
    - `firestore.rules` 以 Firebase Authentication UID 作為資料隔離 key。
- **排除：**
    - 伺服器端 Admin SDK 的讀寫，此類操作不受 Security Rules 限制。
- **利弊：**
    - 規則簡單且由 Firebase 執行，無需維護伺服器；但規則變更需重新部署。

---

#### QuotaBaseline — 免費層配額基準

- **功能：**
    - Firebase 免費層提供的 project 級別 Firestore 讀寫配額。
- **目的：**
    - 定義產品可免費使用的 Firestore 資源上限，作為 App 端 QuotaManagement 的基準參考。
- **做法：**
    - 免費層：50,000 reads / 20,000 writes / day，project 層級；App 端本地計數，即 quotaService.ts，為第一道防線。
- **排除：**
    - 自動升級付費 Firebase 方案。
- **利弊：**
    - 免費層配額對早期用戶數量通常足夠，但高成長期需密切監控並規劃升級。

    - 大幅降低跨平台 IAP 維護成本，但引入第三方依賴，且有額外費用。
